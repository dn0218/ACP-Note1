Hub-and-Spoke 架构

Global Cluster (控制核心) + Workload Cluster (业务负载) ，集中管控，独立扩展。

🔹 Global Cluster 核心组件
Gateway · Auth (SSO+RBAC) · Web Console · Cluster Management · OLM · 内置镜像仓库 · Observability · Cluster Proxy
🔸 Workload Cluster
Kubernetes 环境，由 Global 统一管理，运行隔离的应用负载，自动继承治理策略。

⭐ 关键集成 外部 IdP (OIDC/SAML) · 负载均衡 VIP/DNS · API/CLI 访问

🛡️

高可用 & 自愈机制

  • etcd – 3/5节点 + RAFT协议，3节点容错1故障，支持S3快照
  • ALB (Ingress) – 3副本 + 选主，节点健康检查，自建VIP(Keepalived)
  • Registry + MinIO – 3副本 / 纠删码分布式模式
  • 监控组件 – Prometheus (Thanos去重) / VictoriaMetrics 集群模式
  • 日志组件 – Kafka/ES 分布式多副本
  • 网络(CNI) – Kube-OVN/Calico 高可用控制平面

✅ 每个组件均可冗余部署，Global 集群可管理数十至数百个 Workload 集群，故障自愈 + 流量重定向。

节点预处理 · 绝对必查清单

⚠️ 90%安装故障源于节点未严格遵循要求。以下为 强制/条件必检 项 (参考文档 P41-P51)

🔧 OS & 内核

  • 支持 OS: RHEL 7.8/8.6/8.10/9.6, CentOS 7.6~7.9, Ubuntu 20.04/22.04, Kylin V10 SP3
  • grub 参数: transparent_hugepage=never
  • CentOS 7 额外: cgroup.memory=nokmem
  • 加载内核模块: ip_vs, ip_vs_rr, ip_vs_wrr, ip_vs_sh
  • Kube-OVN 需: geneve, openvswitch 模块
  • 禁用 SELinux/AppArmor，禁用防火墙, 关闭 swap

🌐 网络 & 端口

  • hostname ≤36字符，仅小写字母/数字/.-
  • /etc/resolv.conf 必须存在，禁止包含172开头地址，避免过多search域
  • 控制平面端口不可占用: 2379,2380,6443,10249~10256
  • Kube-OVN 端口: 6641,6642 ; Calico: 179
  • 禁止 /tmp 挂载 noexec，删除冲突包 (docker, podman, containerd, kubelet等)
  • 时间同步误差 ≤10s，所有节点 hostname 唯一
# 快速预检脚本示例 (在安装包 res/init.sh)
bash init.sh # 执行基础配置，仍需人工复核上述核心项

📌 特别提醒: 节点必须安装 ip,ss,tar,swapoff,modprobe,sysctl,lvm2(如需TopoLVM) ，并删除 /var/lib/docker /var/lib/containerd 等残留。

ac CLI · 核心操作

ACP CLI 提供类 kubectl 体验，原生多集群会话管理

ac login https://acp.domain --name prod
ac config get-sessions
ac config use-session prod
ac config use-cluster workload-a
ac namespace my-app
ac get pods -n my-ns
ac logs deployment/nginx -f
ac describe pod <pod>
ac adm upgrade --cluster=global --to-latest
ac adm upgrade status --cluster=workload
ac logout --session prod
💡 提示: ac 自动路由全局资源 (projects, users) 至 Global Cluster；日常管理配合 kubectl 完全兼容，并支持多平台 session 隔离。

📌 升级黄金规则
✅ 先升级 Global Cluster 到目标版本 (例如 4.3)
✅ 再升级 Workload Cluster ，前提是 Workload 必须处于兼容 K8s 版本范围内 (1.31~1.34)
✅ 使用 CVO-based 工作流：运行 upgrade.sh --preflight，检查通过后通过 Web Console 或 ac adm upgrade 执行。
⚠️ Global DR 环境需先升级 standby 集群，再升级 primary，最后重新启用 etcd 同步插件。

ACP Core + Extensions 扩展模型

🎯 ACP Core

最小交付单元：集群管理、容器编排、项目/用户管理、最高安全标准、最长支持生命周期。

核心能力 不可缩减，生产必备。

🔌 Extensions (Operator / Cluster Plugin)

Aligned – 随 ACP 版本生命周期对齐，统一升级。
Agnostic – 独立于 ACP 发布，可单独升级（例如 GitOps, Kiali, AI 插件）。

支持 Operator Lifecycle Manager (OLM) 和 集群插件市场，动态表单配置。

安装核心环节

  • 解压 core 包: tar -xvf ACP-xxx.tar.gz -C /root/cpaas-install
  • 启动 installer: bash setup.sh [--ip-family ipv6] (Web UI 配置 VIP, 集群参数)
  • 参数必设: Kubernetes版本, 集群网络协议(dual-stack), Cluster Endpoint (VIP/域名), 镜像仓库凭证
  • VIP 要求: 与所有控制节点同子网，VRRP 可用，不可使用自建VIP时若没有硬件LB
  • 安装后验证: kubectl get apprelease -A 以及 Pod 全部 Running/Completed
🚀

升级工作流 (CVO)

  • 准备升级产物: bash upgrade.sh 同步镜像
  • 预检: bash upgrade.sh --preflight 检查 ResourcePatchUpgradeable 等项
  • 应用升级: ac adm upgrade --cluster=global --to=4.3.0 或 Web Console 两步确认
  • 升级后安全加固: 禁用 PKCE plain 方法 (仅保留 S256) 提升安全性
  • 注意: 所有 workload 集群必须在 global 升级前处于兼容版本 (例如 1.31~1.34)
